# v1.0.5.x 阶段总结与治理落地

> 目标：不止复盘，还把复盘转成可执行治理动作。  
> 对应工具：`npm run governance:report`

---

## 1) 阶段结论（v1.0.5.1 当前）

- **做对了**
  - 版本桥接路径清晰：`1.0.4.4` 平滑进入 `1.0.5.x`。
  - 运维门禁稳定：`go-live:preflight`、`rc:live-spotcheck` 已形成闭环。
  - 体验补丁有效：MCP smoke、索引中的 `@` 保护、Release 链接对齐。
- **仍有风险**
  - 文档体量持续增大，查找与维护成本上升。
  - 脚本数量增加后，存在“可用但不易发现”的治理工具。
  - 部分核心文件偏大，未来迭代会抬高回归成本。

---

## 2) 这轮新增的“真正有用”能力

### A. 文件分类（自动）

- 新增脚本：`scripts/governance-report.mjs`
- 新增命令：`npm run governance:report`
- 自动生成：
  - `docs/reports/FILE_CLASSIFICATION_LAST.md`
  - `docs/reports/CODE_MANAGEMENT_LAST.md`

### B. 代码管理（自动信号）

报告会自动给出：
- 分类统计（前端/后端/API/文档/脚本/测试等）
- 大文件热点（默认 >= 700 行）
- TODO/FIXME/HACK 密度 Top 列表
- 未直接挂在 `package.json scripts` 的脚本清单
- 源码/测试文件比（粗指标）

## 4) 从总结到行动（v1.0.5.3 / v1.0.5.4）

- **文件分类**
  - 把旧世代文档归档到 `docs/archive/<version-line>/`（先 1.0.2.x、1.0.3.x）。
  - `docs/ROADMAP.md` 保留导航，减少历史细节堆叠。
- **代码管理**
  - 对热点大文件前 3 名做边界拆分（函数级/服务级）。
  - 对 TODO Top 10 建 issue 并标记目标版本。
  - 将高频治理命令统一纳入发布前检查清单。

## 5) Done 定义（治理）

- 每次小版本发版前至少跑 1 次 `governance:report`。
- 报告中“热点文件数量”和“高密度 TODO 文件数”趋势不恶化。
- 文档导航页能在 2 跳内定位到当前执行文件。
